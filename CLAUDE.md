# CLAUDE.md

## Project: claude-arcade

A terminal Minesweeper game that auto-launches when Claude Code is working and screams when Claude needs your attention. Built to solve "the reels trap" — the developer habit of context-switching to Twitter/TikTok while Claude is thinking, then missing permission prompts and losing 20+ minutes.

The core insight: a game running in a tmux pane keeps the user's attention *inside the terminal*, so when Claude needs permission, the user actually sees it. The game pauses and flashes red on permission prompts, making the alert impossible to ignore.

## What you are building

A Rust CLI binary (`claude-arcade`) plus a set of Claude Code hook scripts that orchestrate the game's lifecycle.

The user installs once. From then on, whenever Claude Code starts a tool call, a Minesweeper game opens in a split tmux pane. The game shows live status from Claude (working / waiting for permission / idle / done). When Claude finishes the turn, the pane closes.

## Architecture

```
┌─────────────────────────────────────────────────────────────┐
│ User's terminal (tmux session)                              │
│                                                             │
│ ┌─────────────────────────┬─────────────────────────────┐  │
│ │ claude (main pane)      │ claude-arcade (side pane)   │  │
│ │                         │                             │  │
│ │ > write me a script     │   MINESWEEPER  ●●●  Score:42│  │
│ │ ⏺ working...            │   ┌───────────────────────┐ │  │
│ │                         │   │ 1 . . 2 # . . . . . . │ │  │
│ │ [hook fires →]          │   │ . . 1 . . 1 . . . . . │ │  │
│ │                         │   └───────────────────────┘ │  │
│ │                         │   ⏺ Claude is working...    │  │
│ └─────────────────────────┴─────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
                    ▲                        ▲
                    │                        │
              hooks write              game reads state
              to ~/.claude-arcade/     from state file
              state.json               every 100ms
```

### Components

1. **`claude-arcade` binary** (Rust + ratatui) — the game itself. Renders Minesweeper, polls a state file at `~/.claude-arcade/state.json` to know what Claude is doing, reacts visually to state changes.

2. **Hook scripts** (POSIX shell, in `hooks/`) — small scripts that Claude Code invokes. They write JSON to the state file. They are kept tiny on purpose — no Rust, no compiled dependencies, just `jq` and `echo`.

3. **`claude-arcade install`** subcommand — sets up everything: writes hook scripts to `~/.claude/hooks/`, patches `~/.claude/settings.json` to register them, creates the state directory. One-command install is the entire UX promise.

### Why this split

The hooks must be near-instant (Claude blocks on them). A Rust binary startup, even tiny, would add latency to every tool call. Shell scripts that just write a JSON file take ~5ms. The game itself can be Rust because it runs once per session, not per hook event.

## Hook events and what they mean

These are real Claude Code hook events as of November 2025. Do not invent new ones.

| Event | When it fires | What we do |
|-------|---------------|------------|
| `SessionStart` | Claude session begins | Launch tmux pane with `claude-arcade play` |
| `PreToolUse` | Before any tool call | Write `{"status":"working","tool":"<name>"}` to state |
| `Notification` (matcher: `permission_prompt`) | Claude needs permission | Write `{"status":"permission_needed"}` — game flashes red, rings bell, freezes score |
| `Notification` (matcher: `idle_prompt`) | Claude is idle waiting | Write `{"status":"idle"}` — soft yellow border |
| `Stop` | Claude finishes turn | Write `{"status":"done"}` — game shows "Claude is done" banner, score saves |
| `SessionEnd` | Session terminates | Kill the tmux pane |

The hook scripts receive JSON on stdin. They must read it, extract fields with `jq`, and write to the state file. They must always exit 0 — the game is a side effect and must never block Claude.

No tokio. Minesweeper does not need async — a synchronous event loop with `crossterm::event::poll(Duration::from_millis(100))` is correct here. The 100ms poll interval doubles as our state-file refresh tick.

## Event loop pattern

Use the centralized polling pattern from the ratatui book. Pseudocode:

```rust
loop {
    // 1. Poll for keyboard input (100ms timeout = our tick rate)
    if event::poll(Duration::from_millis(100))? {
        match event::read()? {
            Event::Key(key) => app.handle_key(key),
            Event::Resize(_, _) => {} // ratatui handles this in draw()
            _ => {}
        }
    }

    // 2. Refresh Claude state from disk (cheap — small JSON file)
    app.refresh_claude_state();

    // 3. Render
    terminal.draw(|f| render::ui(f, &app))?;

    if app.should_quit { break; }
}
```

Do **not** use async/tokio. Do **not** spawn a separate event thread with mpsc — overkill for a Minesweeper-sized app.

## State file schema

`~/.claude-arcade/state.json`:

```json
{
  "status": "working" | "permission_needed" | "idle" | "done" | "unknown",
  "tool": "Bash" | "Read" | "Edit" | null,
  "message": "human-readable string from Claude (Notification events only)",
  "updated_at": "2026-05-22T14:30:00Z"
}
```

The game watches `updated_at` to detect changes and trigger transitions (flash, bell, etc).

## Visual states (the whole point of the project)

Each Claude status maps to a visual treatment on the game pane. This is the product. Get this right.

- **`working`** — normal Minesweeper, subtle blue border, footer shows `⏺ Claude is working: <tool>`
- **`permission_needed`** — game freezes mid-action, red flashing border, terminal bell rings once, footer shows `⚠  CLAUDE NEEDS PERMISSION — SWITCH PANES`, score multiplier paused
- **`idle`** — yellow border, footer shows `⏸  Claude is waiting for your input`
- **`done`** — green border for 3 seconds, footer shows `✓ Claude finished`, then game continues normally
- **`unknown`** (no state file yet) — game runs without any Claude UI, just plain Minesweeper

The "score multiplier paused on permission_needed" is the gamification hook: missing a permission prompt costs the player score. This is the shareable insight.

## Minesweeper rules (for reference)

Standard rules. Default board: 16×16, 40 mines. Difficulty flag: `claude-arcade play --difficulty easy|medium|hard`.

- Arrow keys / hjkl to move cursor
- Space or Enter to reveal
- `f` to flag
- `r` to restart
- `q` to quit (but game auto-quits on SessionEnd hook anyway)

First reveal is always safe (standard Minesweeper rule — generate the board *after* the first click, ensuring no mine at that position).

## Installation UX (non-negotiable)

**Users must not need Rust installed.** This is the single most important UX constraint. Most Claude Code users are not Rust developers — requiring `cargo install` would gate this on a 1.5GB toolchain install and lose 90% of potential users. We ship pre-built binaries.

### The user-facing install command

```bash
curl --proto '=https' --tlsv1.2 -LsSf https://github.com/<owner>/claude-arcade/releases/latest/download/claude-arcade-installer.sh | sh
```

That single command:
1. Detects OS + arch (macOS Intel/ARM, Linux x86_64/ARM64)
2. Downloads the matching pre-built binary from the latest GitHub Release
3. Verifies the SHA-256 checksum
4. Installs to `~/.local/bin/claude-arcade` (or `/usr/local/bin` if writable, falling back if not)
5. Adds `~/.local/bin` to PATH if needed (only with user consent — respect `CLAUDE_ARCADE_NO_MODIFY_PATH`)
6. Prompts the user to run `claude-arcade install` to wire up the Claude Code hooks

Steps 1–5 are **generated by cargo-dist**, not hand-written. Step 6 is our own subcommand.

### Release pipeline: cargo-dist

Use `cargo-dist` (axodotdev/cargo-dist, v0.31+ as of Feb 2026) to handle the entire release pipeline. It:
- Generates the installer shell script
- Builds binaries for all targets in GitHub Actions
- Publishes to GitHub Releases with checksums
- Updates Homebrew taps if configured

Initial setup (run once):
```bash
cargo install cargo-dist
cargo dist init
```

`cargo dist init` will write the relevant `[workspace.metadata.dist]` block to `Cargo.toml` and a `.github/workflows/release.yml`. Required targets in our config:
- `aarch64-apple-darwin` (Apple Silicon macOS)
- `x86_64-apple-darwin` (Intel macOS)
- `x86_64-unknown-linux-gnu` (most Linux desktops)
- `aarch64-unknown-linux-gnu` (ARM Linux, including Raspberry Pi and ARM cloud VMs)

Windows is explicitly **not** a target for v1 — see Constraints.

Releases trigger on a git tag matching `v*`:
```bash
git tag v0.1.0
git push --tags
# GitHub Actions builds binaries, publishes release, updates installer.sh
```

### Secondary install paths

These are nice-to-haves, not required for launch:
1. **Homebrew**: `brew install claude-arcade` (cargo-dist can publish a tap automatically)
2. **Cargo**: `cargo install claude-arcade` (kept for Rust devs who prefer it; same binary, slower path)

Document all three in the README in this order: curl install (default), Homebrew (mac), cargo install (rust devs).

### The `claude-arcade install` subcommand

Separate from the binary install above. This is what wires the binary into Claude Code's hooks. It runs *after* the binary is installed.

It must:
1. Check for `tmux` (see Dependency checks below). If missing, print install instructions and exit non-zero.
2. Create `~/.claude-arcade/` directory.
3. Write hook scripts to `~/.claude/hooks/claude-arcade/` (namespaced to avoid clobbering anything).
4. Read existing `~/.claude/settings.json` (or create if missing).
5. Merge our hook entries — **do not clobber the user's existing hooks**. Use the existing `hooks.{event}[]` arrays, append our entries, deduplicate.
6. Show a diff of changes and prompt `Apply these changes? [Y/n]` unless `--yes` is passed.
7. Write `settings.json` atomically (write to tempfile, fsync, rename).
8. Print a success message with next-step instructions.

A `claude-arcade uninstall` subcommand must do the reverse: remove our hook entries from settings.json, delete `~/.claude/hooks/claude-arcade/`, leave `~/.claude-arcade/stats.json` in place (user might reinstall).

### Dependency checks

The install script and `claude-arcade install` must both check for `tmux` and refuse cleanly if missing. Do **not** attempt to install tmux automatically — that's a sharp-edge move that breaks the user's package manager state.

Message format:
```
✗ claude-arcade requires tmux, which was not found in PATH.

  Install with:
    macOS:   brew install tmux
    Debian:  sudo apt install tmux
    Fedora:  sudo dnf install tmux
    Arch:    sudo pacman -S tmux

  Then re-run: claude-arcade install
```

### What this means for development

The `Cargo.toml` is still authored normally. `cargo build --release` still works for local testing. cargo-dist only kicks in for releases. The dev loop is unchanged from a normal Rust CLI.

## Constraints and rules

1. **No Rust toolchain requirement for end users.** Distribution is pre-built binaries via cargo-dist. `cargo install` is a secondary path for Rust devs only. If you find yourself writing docs that tell users to install Rust, stop — that's a regression.
2. **Never break Claude Code.** Hooks must always exit 0. If our state file is missing or malformed, the game must degrade to plain Minesweeper, not crash.
3. **Hooks must be fast** — under 50ms. No network calls. No heavy parsing. Just `jq` + `echo`.
4. **No tmux dependency in the game itself.** The game must run standalone (e.g. for development/testing). tmux is only used by the SessionStart hook to launch the pane. If tmux is not present, both the installer and the SessionStart hook should detect this and print a clear install message (see Dependency checks above).
5. **Cross-platform: macOS and Linux only for v1.** Windows is out of scope until v2 — the hook script ecosystem assumes POSIX shell, and tmux's Windows story is messy. Detect platform in the installer and refuse cleanly on Windows with a "coming in v2" message.
6. **No telemetry. No network. Ever.** This is a developer tool that watches Claude's behavior — privacy matters. Be paranoid about this. The only network call in the entire stack is the one-time binary download during install.
7. **Match the existing terminal color scheme.** Use ratatui's default styling that respects `NO_COLOR` and 256-color terminals. Do not hardcode colors that look bad on light themes.

## Things to NOT do

- Do not add more games in v1. Minesweeper only. The plugin system is v2.
- Do not add a leaderboard, network sync, or user accounts. Local stats only (`~/.claude-arcade/stats.json`).
- Do not parse Claude's actual prompts or tool inputs. We only react to lifecycle events. This is both a privacy decision and a simplicity decision.
- Do not add configuration before users ask for it. Difficulty flag and that's it.
- Do not write a custom event loop abstraction. Use the ratatui pattern directly.

## Coding conventions

- Use `color-eyre` for error handling everywhere except library boundaries.
- Functions over 40 lines should probably be split.
- No `unwrap()` in production paths — use `?` or `expect()` with a specific message.
- Tests for board logic (`game/board.rs`) are required. UI rendering tests are not.
- Run `cargo fmt` and `cargo clippy -- -D warnings` before committing.

## How to verify changes

```bash
# 1. Build clean
cargo build --release

# 2. Run game standalone (no Claude integration needed)
cargo run -- play

# 3. Run unit tests for board logic
cargo test

# 4. Test install dry-run
cargo run -- install --dry-run

# 5. End-to-end test: simulate a hook event by writing to the state file manually
echo '{"status":"permission_needed","updated_at":"2026-05-22T14:30:00Z"}' > ~/.claude-arcade/state.json
# Then watch the running game flash red

# 6. Verify the release pipeline locally before pushing a tag
cargo dist build           # builds all target binaries locally
cargo dist plan            # shows what the release would produce
# Only push a v* tag once `cargo dist plan` looks correct
```
